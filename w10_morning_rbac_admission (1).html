02
RBAC

Ai làm gì trong cụm?

Role-Based Access Control: phân quyền cho người/service account.

Khái niệm

RBAC có 4 thứ — hiểu rõ 2 cặp này là xong

Role / ClusterRole

= Định nghĩa quyền
Role → 1 namespace
ClusterRole → toàn cụm
Chưa gắn cho ai = chưa có tác dụng
+

RoleBinding / ClusterRoleBinding

= Gắn quyền cho ai đó
RoleBinding → 1 namespace
ClusterRoleBinding → toàn cụm
Gắn xong → user có quyền thật

Tóm lại: Role = định nghĩa quyền (chưa gắn cho ai). RoleBinding = gắn cho user/SA cụ thể.

02 — RBAC
Ví dụ cụ thể

Alice chỉ được xem pods trong namespace demo

Tạo Role + RoleBinding → xong.

1
Role
apiVersion: rbac.authorization.k8s.io/v1 kind: Role metadata: namespace: demo name: pod-reader rules: - apiGroups: [""] resources: ["pods"] verbs: ["get", "list"] # chỉ xem
2
RoleBinding
apiVersion: rbac.authorization.k8s.io/v1 kind: RoleBinding metadata: namespace: demo name: read-pods subjects: - kind: User name: alice # ← gắn cho alice roleRef: kind: Role name: pod-reader # ← role bên trái
Kết quả:
Alice xem được pods trong demo
Không xem được prod/staging
Không tạo/xóa gì cả
02 — RBAC
Ví dụ cụ thể

Bob là SRE — xem pods toàn cụm

Dùng ClusterRole. Không giới hạn namespace.

1
ClusterRole
apiVersion: rbac.authorization.k8s.io/v1 kind: ClusterRole # toàn cụm metadata: name: cluster-pod-reader # KHÔNG có namespace rules: - apiGroups: [""] resources: ["pods"] verbs: ["get", "list", "watch"]
2
ClusterRoleBinding
apiVersion: rbac.authorization.k8s.io/v1 kind: ClusterRoleBinding metadata: name: sre-read-pods subjects: - kind: User name: bob # ← gắn cho bob roleRef: kind: ClusterRole name: cluster-pod-reader
Chạy được
kubectl get pods --all-namespaces
kubectl get pods -n demo/prod/staging
KHÔNG chạy được
kubectl delete/create/scale ...
(chỉ xem, không sửa)

ServiceAccount

Pod cũng cần quyền

Ví dụ: Prometheus pod cần gọi K8s API để list pods.

Vấn đề: Pod muốn gọi K8s API → cần identity. Dùng ServiceAccount (SA).

1
ServiceAccount
apiVersion: v1 kind: ServiceAccount metadata: { name: app-sa, namespace: prod }
2
Role
apiVersion: rbac.authorization.k8s.io/v1 kind: Role metadata: { name: pod-lister, namespace: prod } rules: - apiGroups: [""] resources: ["pods"] verbs: ["list"]
3
RoleBinding
apiVersion: rbac.authorization.k8s.io/v1 kind: RoleBinding metadata: { name: app-list-pods, namespace: prod } subjects: - kind: ServiceAccount # ← gắn SA { name: app-sa, namespace: prod } roleRef: { kind: Role, name: pod-lister, ... }
4
Pod dùng SA
# trong Deployment/Pod: spec: serviceAccountName: app-sa # ← xài SA containers: - { name: prometheus, image: prom/prometheus } # giờ gọi K8s API với quyền của app-sa
Kết quả:
Pod có quyền list pods
Không gắn SA → 403 Forbidden

Test RBAC

kubectl auth can-i — test quyền ngay lập tức

Tạo Role/RoleBinding xong → test ngay.

Test chính mình
# Tôi tạo pod được không? kubectl auth can-i \ create pods -n demo yes
Test user khác (--as)
# Alice có quyền không? kubectl auth can-i \ delete deployments -n prod \ --as alice no
Test ServiceAccount
# SA có list được không? kubectl auth can-i \ list pods -n prod --as \ system:serviceaccount:prod:app-sa yes

Dùng khi nào?

  • Sau khi tạo RBAC → test trước khi submit
  • User báo "không có quyền" → reproduce lỗi
  • Pod bị 403 → test quyền của SA
  • Debug RBAC lúc 2h sáng production down

03
Admission Policy

Chặn manifest xấu TẠI API server

RBAC = ai làm gì. Admission = manifest như thế nào.

Khái niệm

RBAC chỉ kiểm "ai" — chưa kiểm "cái gì"

Alice được tạo Deployment (RBAC pass). Nhưng Deployment đó có image lạ, không có limits → cần chặn ở đây.

Flow khi bạn kubectl apply (4 bước kiểm tra)
Bước 1
Authentication
Bạn là ai?
→
Bước 2
Authorization (RBAC)
Bạn có quyền không?
→
Bước 3
Admission Webhooks
Manifest có hợp lệ không?
→
Bước 4
Persisted to etcd
Lưu vào cluster
Ví dụ pass

Alice (đã auth, có RBAC) deploy manifest có image từ ECR, có limits, có label owner. Policy check → pass → vào etcd.

Ví dụ bị chặn

Alice deploy manifest không có limits. RBAC pass (alice có quyền), nhưng Admission webhook reject → trả lỗi 403, manifest KHÔNG vào cụm.

Tóm lại:
RBAC kiểm "ai làm gì"
Admission Policy kiểm "cái gì như thế nào" (image từ đâu? có limits không? có label không?)
→ Kết hợp 2 lớp = cluster thật sự an toàn
03 — Admission Policy
Gatekeeper

OPA Gatekeeper = Policy-as-Code

Viết logic 1 lần, áp cho cả cluster. Không phải gõ tay từng rule.

Gatekeeper có 2 thứ:

1
ConstraintTemplate

= Khuôn mẫu policy (viết logic bằng Rego).
Ví dụ: "Deployment phải có label X".
Định nghĩa 1 lần, dùng nhiều lần.

2
Constraint

= Instance cụ thể của template.
Truyền tham số (label nào? resource nào?)
Chọn enforce/audit. Áp policy lên cluster.

Audit mode vs Enforce mode

Audit (enforcementAction: warn):
Report vi phạm, KHÔNG chặn. Dùng khi test policy mới.

Enforce (enforcementAction: deny):
Reject ngay lập tức. Dùng khi policy đã stable.

spec: enforcementAction: deny # enforce # hoặc: warn (audit)
03 — Admission Policy
Gatekeeper

ConstraintTemplate — định nghĩa khung policy

Viết logic Rego 1 lần → sinh ra CRD mới. Constraint dùng CRD đó để áp lên cluster.

1
ConstraintTemplate (logic Rego)
# Template: policy "phải có label owner" apiVersion: templates.gatekeeper.sh/v1 kind: ConstraintTemplate metadata: { name: k8srequiredlabels } spec: crd: spec: names: { kind: K8sRequiredLabels } # sinh ra CRD mới tên này validation: openAPIV3Schema: properties: labels: { type: array } # param nhận vào từ Constraint targets: - target: admission.k8s.gatekeeper.sh rego: | package k8srequiredlabels violation[{"msg": msg}] { provided := {l | input.review.object.metadata.labels[l]} required := {l | l := input.parameters.labels[_]} missing := required - provided count(missing) > 0 # thiếu label → vi phạm msg := sprintf("Thiếu label: %v", [missing]) }
03 — Admission Policy
Ví dụ

Constraint — áp policy "owner label bắt buộc"

# Constraint: instance của template trên apiVersion: constraints.gatekeeper.sh/v1beta1 kind: K8sRequiredLabels # dùng CRD từ template metadata: { name: require-owner-label } spec: enforcementAction: deny # deny = enforce, warn = audit match: kinds: - apiGroups: ["apps"] kinds: ["Deployment"] parameters: labels: ["owner"] # truyền param vào template
▶ Test vi phạm
# Deployment thiếu label owner kubectl apply -f deploy-no-owner.yaml Error from server (Forbidden): admission webhook denied the request: Missing labels: {"owner"} # Fix: thêm label owner metadata: labels: owner: team-platform # → apply thành công
Flow đầy đủ:
• ConstraintTemplate = viết logic Rego, sinh ra CRD K8sRequiredLabels
• Constraint = dùng CRD đó, truyền labels: ["owner"], chọn deny/warn
• Gatekeeper webhook reject manifest vi phạm

Lab 1.2 · Gatekeeper

Đề bài: 4 luật chặn manifest xấu tại admission

Cài OPA Gatekeeper (qua GitOps) + viết 4 constraint. Deploy vi phạm phải bị API server từ chối.

#Luật cần enforceRisk
1Cấm image tag :latestF-01
2Bắt buộc có resources.limitsF-02
3Cấm runAsUser: 0 (chạy root)F-04
4Cấm hostNetwork: true—

Gợi ý — file & thứ tự

  • Cài controller: argocd/apps/gatekeeper.yaml (sync-wave sớm)
  • ConstraintTemplate (Rego) lấy từ gatekeeper-library — không cần tự viết Rego
  • gatekeeper/constraints/ chứa 4 Constraint + 1 App quản
  • Thứ tự: controller → ConstraintTemplate → Constraint (dùng sync-wave, đừng apply 1 lượt)
Bẫy: chính Rollout api của bạn có lọt 4 luật không? Kiểm tra image đã pin version, có limits, không set runAsUser:0 — nếu platform tự bị chặn thì sửa cho hợp lệ trước khi bật enforce.
